# Разбор ДЗ: рефакторинг теста регистрации с Page Object

Ветка: `refactor_userShouldBeRegistredAsMale_test_with_page_object`
Автор исходного кода: Мария Сатарова
Файлы под разбором: `DemoQaRegistrationPage.java`, `WelcomePage.java`, `QaTest.java`

Ниже — что я поправил в твоём коде и зачем. Это не «код был плохой», а следующий шаг по чистоте: ты уже сделала главное — вынесла логику в Page Object. Правки касаются того, **как именно** методы страницы общаются с тестом.

---

## Главная идея всех правок

**Тест не должен знать про внутренности страницы.** Он должен читаться как сценарий на человеческом языке, а не как набор технических команд.

Сравни, как выглядел вызов в тесте до и после:

```java
// БЫЛО — тест знает про id полей и про строку "Male"
.setFieldById("firstName", firstName)
.setFieldById("lastName", lastName)
.setFieldById("userNumber", phoneNumber)
.setRadioButtonByValue("Male")

// СТАЛО — тест говорит ЧТО делает, а не КАК
.setFirstName(firstName)
.setLastName(lastName)
.setPhoneNumber(phoneNumber)
.setMaleGender()
```

`"firstName"`, `"userNumber"`, `"Male"`, `[value=%s]` — это **детали реализации страницы**. Если завтра разработчики переименуют id поля или поменяют селектор радиокнопки, чинить нужно **одно место в Page Object**, а не лазить по всем тестам. Сейчас тест от этих деталей полностью отвязан.

Это и есть инкапсуляция применительно к Page Object: наружу — понятные доменные методы, внутрь — спрятанные технические хелперы.

## 1. `DemoQaRegistrationPage.java`

### 1.1. Универсальные методы стали приватными хелперами

```java
// было public — тест мог звать напрямую с "магическими" строками
public DemoQaRegistrationPage setFieldById(String fieldName, String fieldValue) { ... }
public DemoQaRegistrationPage setRadioButtonByValue(String value) { ... }

// стало private — это внутренняя кухня страницы
private DemoQaRegistrationPage setFieldById(String fieldName, String fieldValue) { ... }
private DemoQaRegistrationPage setRadioButtonByValue(String value) { ... }
```

**Зачем:** эти методы удобные, их не нужно удалять. Но они — инструмент *внутри* страницы. Тест ими пользоваться не должен, поэтому модификатор `private`. Я их перенёс вниз класса — публичный «интерфейс» страницы сверху, приватная реализация снизу. Так класс читается как оглавление.

### 1.2. Появились доменные методы-обёртки

```java
public DemoQaRegistrationPage setFirstName(String firstName)  { return setFieldById("firstName", firstName); }
public DemoQaRegistrationPage setLastName(String lastName)     { return setFieldById("lastName", lastName); }
public DemoQaRegistrationPage setPhoneNumber(String phoneNumber){ return setFieldById("userNumber", phoneNumber); }
public DemoQaRegistrationPage setMaleGender()                 { return setRadioButtonByValue("Male"); }
public DemoQaRegistrationPage setFemaleGender()              { return setRadioButtonByValue("Female"); }
```

**Зачем:** каждое поле формы получило свой осмысленный метод. Знание «телефон лежит в поле `userNumber`» теперь живёт ровно в одном месте. В тесте — только бизнес-смысл.

Обрати внимание на `setMaleGender()` / `setFemaleGender()` — у них **нет параметра**. Пол — это не произвольная строка, которую передаёт тест, а конкретный выбор. Метод без аргументов невозможно вызвать неправильно (нельзя передать опечатку `"male"` или `"Man"`). Это и надёжнее, и читается лучше: `.setMaleGender()`.

### 1.3. Переименование методов: убрали «click»/«check», добавили смысл

| Было | Стало | Почему |
|------|-------|--------|
| `clickSubmitButton()` | `submit()` | Тесту важно действие «отправить форму», а не то, что технически это клик по кнопке. |
| `clickDateOfBirthField()` | `dateOfBirth()` | Короче, читается как шаг сценария. |
| `chooseDateOnTheTable(...)` | `chooseDate(...)` | «OnTheTable» — деталь верстки (календарь нарисован таблицей). Тесту это не нужно. |

**Зачем:** имя метода описывает **намерение пользователя**, а не способ реализации в DOM. Префиксы `click...` привязывают название к текущей вёрстке; если кнопку заменят на что-то другое, имя `submit()` останется верным, а `clickSubmitButton()` начнёт врать.

### 1.4. Одна проверка разбита на две

```java
// БЫЛО — один метод проверяет сразу две вещи
public DemoQaRegistrationPage checkRegistrationSuccessModalTableVisible() {
    registrationSuccessModal.shouldBe(visible);
    registrationInformationTable.shouldBe(visible);
    return this;
}

// СТАЛО — два отдельных метода, каждый про своё
public DemoQaRegistrationPage verifyModalAppeared() {
    registrationSuccessModal.shouldBe(visible);
    return this;
}
public DemoQaRegistrationPage verifyInformationTableAppeared() {
    registrationInformationTable.shouldBe(visible);
    return this;
}
```

**Зачем:**
- **Одна проверка — одна ответственность.** Если упадёт, по имени метода сразу видно, что именно не появилось: модалка или таблица. В старом варианте при падении непонятно, какая из двух проверок не прошла.
- **Имя стало честным.** Старое `checkRegistrationSuccessModalTableVisible` обещало проверить «модалку-таблицу» как одну сущность, хотя проверяло две разные.
- **`verify` вместо `check`** — единый глагол для проверок-ассертов, отличающий их от методов-действий.

В тесте теперь это два читаемых шага подряд:
```java
.verifyModalAppeared()
.verifyInformationTableAppeared()
```

## 2. `WelcomePage.java`

Та же логика, что и в п.1: универсальный `clickButton(String)` спрятан в `private`, а наружу выставлены два доменных метода.

```java
public WelcomePage wannaBeQa() {
    return clickButton("Хочу вкатиться в QA");
}

public LavaTopPayingPage readyToPay() {
    clickButton("Бегу оплачивать");
    return new LavaTopPayingPage();
}

private WelcomePage clickButton(String buttonText) {
    $(byText(buttonText)).click();
    return this;
}
```

**Зачем:** тексты кнопок («Хочу вкатиться в QA», «Бегу оплачивать») — это контент страницы. Тесту не нужно их знать, ему нужны действия `wannaBeQa()` и `readyToPay()`.

**Отдельно — самый важный момент здесь:** метод `readyToPay()` возвращает **`LavaTopPayingPage`**, а не `this`.

Это правильное моделирование навигации в Page Object: клик по «Бегу оплачивать» уводит на страницу оплаты, поэтому метод и **возвращает следующую страницу**. Благодаря этому в тесте получается непрерывная «текучая» цепочка, и переход между страницами происходит сам собой — об этом ниже.

## 3. `QaTest.java`

### 3.1. URL вынесены в константы

```java
private static final String PRACTICE_FORM_URL = "https://demoqa.com/automation-practice-form";
private static final String YANDEX_URL = "https://ya.ru/";
```

**Зачем:** адрес перестал быть «магической строкой» внутри `open(...)`. У него появилось имя, его видно сверху, и если он используется не раз — менять в одном месте. Плюс константа документирует, *что* за URL открывается.

### 3.2. Почищены импорты

```java
// было
import ru.bulgakov.pages.LavaTopPayingPage;
import static com.codeborne.selenide.Condition.*;
import static com.codeborne.selenide.Selenide.*;

// стало
import static com.codeborne.selenide.Selenide.open;
```

**Зачем:**
- `LavaTopPayingPage` в тесте больше не упоминается напрямую (см. ниже) — импорт убран.
- Точечный импорт `Selenide.open` вместо `Selenide.*` и `Condition.*`. Теперь по импортам видно, что тест из Selenide использует **только** `open()`. Все условия (`visible`, `exactText` и т.д.) теперь живут внутри Page Object — а это ровно то, чего мы добивались: **тест не оперирует селенайдовскими проверками напрямую.** Если в тесте снова появился бы `import Condition.*`, это был бы сигнал, что какая-то проверка «утекла» из страницы в тест.

### 3.3. Убрана разорванная цепочка вызовов

```java
// БЫЛО — отдельная переменная и проверка в самом конце, оторванная от сценария
LavaTopPayingPage lavaTopPayingPage = new LavaTopPayingPage();

open("https://ya.ru/", YandexSearchPage.class)
        ...
        .clickButton("Хочу вкатиться в QA")
        .clickButton("Бегу оплачивать");

lavaTopPayingPage.checkPrice("47 000.00");

// СТАЛО — единая непрерывная цепочка
open(YANDEX_URL, YandexSearchPage.class)
        ...
        .wannaBeQa()
        .readyToPay()
        .checkPrice("47 000.00");
```

**Зачем:** раньше объект `LavaTopPayingPage` создавался руками заранее и висел отдельной переменной, а финальная проверка цены была оторвана от основного сценария. Из-за этого терялась связность: глазами надо было «склеивать» две части.

Теперь, когда `readyToPay()` сам возвращает `LavaTopPayingPage`, цепочка не рвётся: переход на страницу оплаты и проверка цены — единый поток. Лишняя переменная не нужна, ручное `new LavaTopPayingPage()` ушло. Тест читается как один связный сценарий сверху вниз.

### 3.4. Тест переведён на новые методы

Оба теста теперь зовут доменные методы из пунктов 1 и 2. Итоговый сценарий регистрации читается почти как обычный текст:

```java
open(PRACTICE_FORM_URL, DemoQaRegistrationPage.class)
        .registrationFormIsVisible()
        .setFirstName(firstName)
        .setLastName(lastName)
        .setPhoneNumber(phoneNumber)
        .setMaleGender()
        .dateOfBirth()
        .chooseDate("12", "February", "1980")
        .submit()
        .verifyModalAppeared()
        .verifyInformationTableAppeared()
        .checkFullNameSavedCorrecly(firstName, lastName);
```

## Что обсудить с ученицей дополнительно

Это не ошибки в моих правках, а точки роста на будущее — стоит проговорить:

1. **Единый стиль имён у проверок.** Я ввёл `verifyModalAppeared()` / `verifyInformationTableAppeared()` с глаголом `verify`, но в классе осталось `registrationFormIsVisible()` и `checkFullNameSavedCorrecly()`. Хорошая привычка — выбрать **один** префикс для всех ассертов (`verify...` либо `check...`) и держаться его во всём проекте. Сейчас три разных стиля.

2. **Опечатка в имени метода: `checkFullNameSavedCorrecly` → `checkFullNameSavedCorrectly`.** Пропущена буква «t» в «Correctly». Я намеренно её не трогал — оставил тебе как повод показать ученице, что имена методов тоже вычитывают, а IDE умеет переименовывать безопасно (Rename, не поиском-заменой).

3. **`setFemaleGender()` пока нигде не вызывается.** Это нормально (метод про запас, симметрия с `setMaleGender`), но стоит проговорить принцип: код пишем под потребность. Если метод нужен для будущих тестов — ок; если «на всякий случай» — лучше добавить тогда, когда появится тест.

4. **Имена-существительные у действий: `dateOfBirth()`, `chooseDate()`.** `chooseDate()` — глагол, хорошо. А `dateOfBirth()` — существительное, по имени не считывается действие (открыть календарь). Можно обсудить вариант вроде `openDateOfBirthPicker()`. Это вкусовщина, но повод поговорить про «метод = действие, поле = существительное».

## Короткое резюме для разбора

Что мы отрабатываем этим рефакторингом:
- **Инкапсуляция в Page Object:** доменные методы — `public`, технические хелперы и селекторы — `private`.
- **Имена по намерению, а не по реализации:** `submit()`, а не `clickSubmitButton()`.
- **Single Responsibility:** одна проверка — один метод.
- **Навигация через возврат следующей страницы:** `readyToPay()` → `LavaTopPayingPage`, отсюда непрерывная цепочка.
- **Чистота теста:** никаких магических строк, лишних импортов и оторванных переменных — тест читается как сценарий.
